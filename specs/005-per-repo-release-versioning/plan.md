# Implementation Plan: Per-Repo Release Versioning

**Branch**: `006-per-repo-release-versioning` | **Date**: 2026-05-23 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `specs/005-per-repo-release-versioning/spec.md`

## Summary

Teams ship different repositories at different cadences within a single project release. This feature changes release composition so that each included repository carries its own independently-set next version and an immutable historical snapshot (previous version, commit range, change counts). The release-level version label is derived automatically from the primary repository's confirmed next version.

Key technical additions: a `ReleaseRepository` join entity with snapshot columns, a new `IReleaseCompositionService` that orchestrates multi-repo preview / create-draft / update-draft flows, a per-repo call to the existing `IVersionBumpService`, six new REST endpoints under `/api/v1/projects/{projectId}/releases`, and three new frontend components (releases list tab, wizard step, per-repo detail table).

## Technical Context

**Language/Version**: .NET 10 (backend), React 18 + TypeScript (frontend)  
**Primary Dependencies**: EF Core 10 + SQLite, ASP.NET Core Web API, FluentValidation, Mapster, TanStack Query, shadcn/ui, React Hook Form + Zod  
**Storage**: SQLite WAL — `./data/repomanager.db`  
**Testing**: xunit + FluentAssertions + Moq (unit), SQLite per-test file (integration), Playwright (E2E, if present)  
**Target Platform**: Web — browser (React SPA) + Linux server (.NET)  
**Project Type**: Full-stack web application  
**Performance Goals**: Release detail page (20 repos) loads in < 2 s; releases list (100 releases) loads in < 2 s  
**Constraints**: Snapshot fields immutable post-publish; no external API calls from request handlers; server always re-derives snapshot, never trusts client-supplied values  
**Scale/Scope**: Up to 100 releases per project; up to 20 repos per release

## Implementation Notes

### Backend sequence

1. **Domain entity first** — `ReleaseRepository` POCO and nav property on `Release`. No EF config yet.
2. **TDD gate** — write failing unit tests for `DeriveReleaseVersion` (primary included, primary excluded, single repo) and `VersionBumpService.SuggestAsync` (no commits, only fixes, mixed feat/fix, breaking footer, breaking `!` marker) before any service code.
3. **IVersionBumpService** — interface in Application, implementation in Infrastructure, DI registration.
4. **IReleaseCompositionService** — interface in Application, implementation in Infrastructure, DI registration.
5. **EF Core configuration** — `ReleaseRepositoryConfiguration` with unique index, cascade/restrict delete.
6. **Migration** — `AddReleaseRepository` with backfill SQL (see `data-model.md`). Idempotent backfill required.
7. **Controller** — `ReleasesController` with 6 endpoints, FluentValidation wired in.
8. **Integration tests** — full flow and backfill migration coverage.

### Frontend sequence

9. Regenerate OpenAPI client after backend is runnable.
10. `<ProjectReleasesList />` tab on the project detail page.
11. `<ReleaseRepoSelectionStep />` wizard step.
12. `<ReleaseRepositoriesTable />` section on the release detail page.

### Critical invariants

- Server **always re-derives** snapshot fields via `IVersionBumpService.SuggestAsync`; client-supplied values are discarded.
- `Release.Version` is **never user-editable**; always set by `DeriveReleaseVersion` inside the service.
- `UpdateDraftAsync` throws `ConflictException(release_not_draft)` if `Status != Draft`. Controller maps to `409`.
- Backfill migration must be idempotent — running it twice must not produce duplicates.
- Legacy release rows (backfilled) display em-dash placeholders and a "Pre-feature release — partial data" badge in the UI.
